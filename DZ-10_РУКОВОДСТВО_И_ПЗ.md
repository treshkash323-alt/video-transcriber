## 1. Зачем этот проект

### 1.1. Задание курса (Lite)

Собрать сервис, который:

- принимает **видео/аudio** через веб-интерфейс;
- **не блокирует** браузер — сразу отдаёт `task_id`;
- обрабатывает файлы **в фоне** (Celery Worker + Whisper);
- показывает статусы **PENDING → STARTED → SUCCESS**;
- масштабируется командой `--scale worker=3`.

### 1.2. Что вы научились делать

| Навык | Где проявился |
|-------|----------------|
| FastAPI | `app/main.py` — upload, status, HTML UI |
| Celery | `app/tasks.py` — задача `transcribe_video` |
| Redis | Очередь задач и хранение результатов |
| Docker Compose | 4 сервиса: web, worker, redis, flower |
| Flower | Мониторинг воркеров и задач на :5555 |
| Whisper | OpenAI API или local faster-whisper |

### 1.3. Связь с ДЗ-9 и будущими проектами

В **ДЗ-9** (CRM) вы делали синхронный веб + Supabase. Здесь — **асинхронная очередь**: тяжёлая работа (транскрибация) уходит в worker. Этот же каркас потом пригодится для:

- **EDU** — транскрипт лекции → урок + RAG;
- **CRM** — webhook с Tilda → очередь обработки;
- **Tilda** — загрузка видео с лендинга без ожидания.

## 2. Глоссарий

| Термин | Простыми словами | В проекте |
|--------|------------------|-----------|
| **FastAPI** | Python-фреймворк для HTTP API | Принимает файлы, отдаёт HTML |
| **Celery** | Система фоновых задач | Worker транскрибирует видео |
| **Redis** | Быстрое хранилище в памяти | Broker + backend для Celery |
| **Worker** | Процесс, который выполняет задачи из очереди | `celery -A app.tasks worker` |
| **Task / task_id** | Одна задача в очереди + её UUID | Возвращается после upload |
| **PENDING** | Задача в очереди, ещё не взята worker | Сразу после upload |
| **STARTED** | Worker начал выполнение | Видно при polling `/status` |
| **SUCCESS** | Задача завершена, есть результат | Текст транскрипта в таблице |
| **Flower** | Веб-UI для Celery | http://localhost:5555 |
| **Whisper** | Модель распознавания речи | OpenAI API или faster-whisper |
| **Broker** | Куда Celery кладёт новые задачи | `redis://redis:6379/0` |
| **Backend** | Где хранятся результаты задач | Тот же Redis |
| **Docker Compose** | Запуск нескольких контейнеров одной командой | `docker compose up --build` |
| **scale worker=3** | Три копии worker — параллельная обработка | Для демонстрации в ДЗ |

## 3. Архитектура

### 3.1. Схема «кто с кем говорит»

```
┌─────────────┐   POST /transcribe    ┌─────────────┐   .delay()    ┌─────────────┐
│  Браузер    │ ────────────────────► │  web        │ ────────────► │  Redis      │
│  :8000      │ ◄── task_id           │  FastAPI    │               │  (очередь)  │
└─────────────┘                       └─────────────┘               └──────┬──────┘
       │                                      │                            │
       │ GET /status/{id}                     │                            │ задача
       └──────────────────────────────────────┘                            ▼
                                                                  ┌─────────────┐
                                                                  │  worker(s)  │
                                                                  │  Whisper    │
                                                                  └─────────────┘
                                                                         │
┌─────────────┐   мониторинг Celery                                      │
│  Flower     │ ◄────────────────────────────────────────────────────────┘
│  :5555      │
└─────────────┘
```

### 3.2. Четыре контейнера

| Сервис | Порт | Роль |
|--------|------|------|
| **web** | 8000 | UI + API, сохраняет файл в `uploads/`, ставит задачу |
| **redis** | только внутри Docker | Очередь и результаты |
| **worker** | — | FFmpeg + Whisper, пишет транскрипт |
| **flower** | 5555 | Дашборд Celery (тот же образ, что worker) |

### 3.3. Почему Flower на том же образе

Раньше использовался готовый образ `mher/flower` — версии Celery/Redis не совпадали с worker, был **Connection refused**. Сейчас Flower собирается из **того же Dockerfile**, что web и worker: одни зависимости, одна версия Celery.

## 4. Первый запуск с нуля

### 4.1. Что нужно на компьютере

1. **Docker Desktop** — Engine running (кит в трее зелёный).
2. **Папка проекта** — `Projects\ДЗ-10\video-transcriber\`.
3. **Один терминал VS Code** — не запускайте `docker compose` в двух местах сразу.

### 4.2. Настройка `.env`

```powershell
cd Projects\ДЗ-10\video-transcriber
copy .env.example .env
```

Откройте `.env` и укажите ключ OpenAI:

```env
TRANSCRIPTION_BACKEND=openai
OPENAI_API_KEY=sk-proj-...
```

Файл `.env` **не попадает в git** (см. `.gitignore`).

### 4.3. Запуск полного стека

```powershell
docker compose up --build
```

Или скрипт:

```powershell
.\scripts\up.ps1
```

Дождитесь строк в терминале:

```
web-1     | Application startup complete
worker-1  | celery@... ready
flower-1  | Visit me at http://0.0.0.0:5555
```

### 4.4. Проверка URL

| URL | Что должно быть |
|-----|-----------------|
| http://localhost:8000 | Форма загрузки + таблица задач |
| http://localhost:8000/docs | Swagger API |
| http://localhost:5555 | Flower — Workers, Tasks |

## 5. Ежедневная работа

### 5.1. Загрузка файлов

1. Подготовьте **1–3 файла** (видео/аudio), **каждый < 25 МБ** (лимит OpenAI Whisper API).
2. **Choose Files** — для трёх файлов: **Ctrl+клик** по каждому.
3. Под полем — список имён и размеров (красным, если > 25 МБ).
4. **Транскрибировать** — в таблице появятся строки PENDING → STARTED → SUCCESS.
5. Можно загружать **по одному** три раза подряд — это тоже засчитывается.

### 5.2. Остановка

```powershell
Ctrl+C
docker compose down
```

Папка `uploads/` на диске сохраняется.

### 5.3. Логи при проблемах

```powershell
docker compose logs worker
docker compose logs flower
```

## 6. Flower и масштаб worker=3

### 6.1. Flower (обязательно для ДЗ)

- Вкладка **Workers** — активные worker-процессы.
- Вкладка **Tasks** — история SUCCESS / FAILURE.

Если :5555 не открывается — вы запустили не полный стек. Нужно:

```powershell
docker compose up --build
```

(все 4 сервиса, не только `web worker redis`).

### 6.2. Масштабирование

```powershell
Ctrl+C
docker compose up --build --scale worker=3
```

Или: `.\scripts\up-scale.ps1`

Загрузите 3 файла — в Flower видно **3 worker**, задачи идут **параллельно**.

## 7. OpenAI vs local Whisper

| Режим | `.env` | Плюсы | Минусы |
|-------|--------|-------|--------|
| **openai** | `TRANSCRIPTION_BACKEND=openai` + ключ | Быстро, качественно | Лимит **25 МБ**, нужен API key |
| **local** | `TRANSCRIPTION_BACKEND=local` | Без оплаты, нет лимита 25 МБ | Медленно на CPU, первый запуск качает модель |

Для файла **> 25 МБ** (например `video_2025-12-31...mp4` ~26 МБ) в режиме openai будет **ERROR 413** — это ожидаемо. Решения:

- сжать/обрезать файл;
- переключиться на `local`;
- в будущем — FFmpeg-сжатие перед API (см. `TODO_ROADMAP.md`).

После смены `.env`:

```powershell
Ctrl+C
docker compose up --build
```

## 9. Частые ошибки

| Симптом | Причина | Решение |
|---------|---------|---------|
| :5555 Connection refused | Flower не запущен или старый образ | `docker compose up --build` (4 серvice) |
| Задача висит PENDING | Worker не работает | `docker compose logs worker` |
| ERROR / 413 | Файл > 25 МБ (OpenAI) | Меньший файл или `local` |
| Port 8000 busy | Старый compose или второй терминал | `docker compose down`, один терминал |
| 3 файла не выбираются | Без Ctrl+клик | Ctrl+клик по каждому или загрузка по одному |
| Ключ на скрине | `.env` попал в кадр | Обрезать скрин, ключ не коммитить |
